## 阶段 1：基础设施迁移（最安全，不影响现有组件）

- [ ] 1.1 检查当前项目的 `src/lib/utils.ts` 是否存在
- [ ] 1.2 如果不存在，从 mindfulness-app-design 复制 `lib/utils.ts` 到当前项目的 `src/lib/utils.ts`
- [ ] 1.3 检查并安装必要的依赖（clsx, tailwind-merge）
- [ ] 1.4 配置路径别名 `@/lib/utils` 在 `vite.config.ts` 中
- [ ] 1.5 配置路径别名 `@/lib/utils` 在 `tsconfig.app.json` 中
- [ ] 1.6 验证 `cn` 函数可以正常导入和使用（运行项目确保无编译错误）

## 阶段 2：组件差异分析（生成对比报告）

- [ ] 2.1 列出两个项目的所有 UI 组件文件
- [ ] 2.2 对每个组件进行差异分析：
  - [ ] 比较文件内容（使用 diff 工具）
  - [ ] 识别样式差异（className、CSS 类）
  - [ ] 识别逻辑差异（props、事件处理、状态管理）
  - [ ] 识别 API 差异（导出的组件、函数）
- [ ] 2.3 生成组件差异报告文件 `component-diff-report.md`，包含：
  - [ ] 完全相同的组件列表（可直接替换）
  - [ ] 仅样式差异的组件列表（安全迁移）
  - [ ] 有逻辑差异的组件列表（需要谨慎处理）
  - [ ] 每个组件的详细差异说明

## 阶段 3：迁移确定安全的组件

- [ ] 3.1 先迁移完全相同的组件（如果有）
- [ ] 3.2 迁移仅样式差异的组件，对于每个组件：
  - [ ] 备份当前组件文件
  - [ ] 保留当前项目的组件逻辑（props、事件处理、状态管理等）
  - [ ] 更新样式相关的代码（className、CSS 类、样式属性）
  - [ ] 确保组件的 API 接口保持不变
  - [ ] 验证组件可以正常编译
  - [ ] 在现有页面中测试组件是否正常工作
- [ ] 3.3 每迁移一个组件后，运行项目确保无错误

## 阶段 4：生成不确定组件的映射表（等待确认）

- [ ] 4.1 对于有逻辑差异的组件，生成详细的对比文档 `component-mapping-review.md`：
  - [ ] 列出组件名称
  - [ ] 显示源项目和当前项目的关键代码差异
  - [ ] 说明差异的影响范围
  - [ ] 提供迁移建议（保留逻辑 vs 更新逻辑）
- [ ] 4.2 等待人工确认每个不确定组件的迁移方案
- [ ] 4.3 根据确认结果更新迁移计划

## 阶段 5：迁移剩余组件（在确认后）

- [ ] 5.1 根据确认的映射表，迁移剩余组件
- [ ] 5.2 对于每个组件：
  - [ ] 按照确认的方案进行迁移
  - [ ] 保留必要的业务逻辑
  - [ ] 更新样式代码
  - [ ] 验证组件功能正常
- [ ] 5.3 每迁移一个组件后，运行项目确保无错误

## 阶段 6：最终验证和清理

- [ ] 6.1 检查所有 UI 组件文件是否都能正常编译
- [ ] 6.2 验证现有页面（CreatePage、RepairPage、MuseumPage）的视觉表现
- [ ] 6.3 确保所有功能逻辑保持不变
- [ ] 6.4 检查是否有样式冲突或破坏性变更
- [ ] 6.5 移除未使用的样式代码
- [ ] 6.6 确保所有导入路径正确
- [ ] 6.7 验证项目可以正常构建和运行（`npm run build`）
