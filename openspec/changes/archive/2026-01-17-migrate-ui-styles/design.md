## Context
当前项目（farewell-museum）已经有完整的 UI 组件库，但样式可能不够美观。源项目（mindfulness-app-design）有设计良好的 UI 组件样式，我们希望将这些样式迁移过来，同时保留当前项目的业务逻辑。

## Goals / Non-Goals
- Goals:
  - 迁移所有 UI 组件的样式代码
  - 保持组件的 API 和 props 接口不变
  - 保留当前项目的所有业务逻辑
  - 确保迁移后项目可以正常编译和运行
- Non-Goals:
  - 不改变组件的功能逻辑
  - 不修改现有页面的业务代码
  - 不引入新的依赖（除非必要，如 clsx、tailwind-merge）

## Decisions
- Decision: 采用分阶段迁移策略
  - 理由：降低风险，确保代码不出错、可反复修改。先迁移基础设施和确定安全的组件，再处理不确定的组件
- Decision: 仅迁移样式，保留逻辑
  - 理由：当前项目的业务逻辑已经完整，只需要改进视觉表现
- Decision: 逐个组件迁移，而不是批量替换
  - 理由：可以更好地控制迁移过程，确保每个组件都正确迁移，每迁移一个就验证一次
- Decision: 生成组件差异报告和映射表
  - 理由：在迁移前先分析差异，识别风险，对于不确定的组件需要人工确认后再迁移
- Decision: 保留组件的 API 接口不变
  - 理由：避免影响现有页面的使用
- Decision: 每迁移一个组件后立即验证
  - 理由：及早发现问题，避免错误累积

## Risks / Trade-offs
- 风险：样式迁移可能影响现有页面的布局
  - 缓解：迁移后需要验证所有页面的显示效果
- 风险：路径别名配置可能缺失
  - 缓解：检查并配置必要的路径别名
- 权衡：样式美观 vs 功能稳定性
  - 选择：优先保证功能稳定性，样式作为增强

## Migration Plan
1. 检查并创建必要的工具函数（lib/utils.ts）
2. 配置路径别名
3. 逐个组件迁移样式
4. 验证所有页面正常工作
5. 测试编译和运行
